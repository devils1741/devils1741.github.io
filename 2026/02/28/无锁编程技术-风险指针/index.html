无锁编程技术--风险指针

发表于2026-02-28|更新于2026-03-01
|浏览量:

背景

在并发编程中，主要问题可以分为三类：正确性问题、线程同步问题和性能问题。
正确性问题，比如竞态条件、内存可见性和指令重排，这些会导致数据不一致。
线程同步问题，比如死锁以及条件变量中的丢失唤醒，需要通过合理的锁顺序和条件检查来避免。
性能问题，比如锁竞争，在高并发下可能成为瓶颈。
在高并发场景下，我们可以使用atomic + memory ordering构建lock-free结构，从而减少锁竞争，提高性能。
在无GC(垃圾回收机制)的语言中，使用lock-free的结构还有一个问题就是内存回收。因为lock-free的结构线程不会被阻塞，无法知道谁在访问某个内存，因此不能随意free内存。
为了解决这个问题，Maged M提出了无锁编程技术–风险指针(Hazard Pointers, HP)
这种思路其实很直白：线程在访问共享对象前先声明自己正在使用它，从而阻止其他线程提前释放该对象。在线程释放节点后，该节点也不会立即被回收而是存储到回收队列中，回收队列定期检查存储的节点，没有线程使用的节点就会被释放

核心数据结构与算法

HPRecType

每个线程使用结构体HPRecType记录正在访问的内存，所有线程的HPRecType构成一个链表。
同时每个线程都有自己的垃圾回收队列rlist，

// Hazard pointer record structure
struct HPRecType {
    NodeType* HP[K]; // 数组，记录线程正在访问的节点， k是一个常数
    HPRecType* Next; // 指向下一个记录结构体
};
// 链表的头节点
HPRecType* HeadHPRec; 
// Per-thread private variables
listType rlist;     // 暂存已经删除但还不能free的节点
integer rcount;     // 回收节点计数

RetireNode routine

回收算法的总体如下。RetireNode 的作用是把已经删除但不能立即释放的节点加入回收队列，并在积累到一定数量后尝试安全回收。
注意进入待回收队列不代表节点一定会被释放，如果有线程仍然使用这些节点，那么节点还是会暂缓释放

RetireNode(node: *NodeType) {
    rlist.push(node);
    rcount++;

    if (rcount >= R) // R就是阈值
        Scan(HeadHPRec);
}

Scan(head: *HPRecType) {

    // Stage 1: Scan HP list and insert non-null values in plist
    plist.init();  // 一个集合，可以是队列，哈希表，数组等。存所有线程当前公开声明的 hazard pointers
    hprec ← head;

    // 遍历所有线程的HP记录（全局扫描）。
    while (hprec ≠ null) {
        // 每个线程最多保护K个指针
        for (i ← 0 to K-1) {
            hptr ← hprec->HP[i];
            if (hptr ≠ null)
                plist.insert(hptr);
        }
        hprec ← hprec->Next;
    }

    // Stage 2: Search rlist
    // 一次性取出待回收链表的所有元素
    tmplist ← rlist.popAll();
    rcount ← 0;

    node ← tmplist.pop();
    while (node ≠ null) {
        // 查看还有没有线程正在使用该节点
        if (plist.lookup(node)) {
            // 如果仍在使用，就保留
            rlist.push(node);
            rcount++;
        }
        else {
            // 没有线程使用直接回收
            PrepareForReuse(node);
        }

        node ← tmplist.pop();
    }
    // 最后释放plist
    plist.free();
}

伪代码描述

作者以无锁FIFO（Michael & Scott Lock-Free Queue (1996)）为例描述如何将HP插入到现有的无锁队列中.
首先是队列的入队和出队
初始版本的伪代码

// Node structure
structure NodeType {
    Data : DataType;
    Next : *NodeType;
}

// Shared variables
Head, Tail : *NodeType;

// Initially both Head and Tail point to a dummy node
// 这里的dummy node是一个始终存储在但是不保存任何数据的节点
// 使得空队列和非空队列具有相同的结构，能够统一并发路径避免了null状态带来的额外复杂的同步操作
Enqueue(data : DataType) {
  node ← NewNode();
  node->Data ← data;
  node->Next ← null;
  // CAS操作可能失败，所以需要while不停的重试
    while true {
      // 获取一个尾节点的快照
      t ← Tail;
      next ← t->Next;
      // 插入新元素后tail完成移动
      if (Tail ≠ t)
            continue;
      // 插入新元素后tail还没有来的及移动
      if (next ≠ null) {
            CAS(&Tail, t, next);
            continue;
        }
      // 确定是尾节点
      if CAS(&t->Next, null, node)
            break;
    }
  CAS(&Tail, t, node);
}

 Dequeue() : DataType
{
    while true {
        // 获取当前头、尾和首个有效节点的快照
        h ← Head;          // h 指向哨兵节点 (Dummy Node)
        t ← Tail;          // t 指向当前的尾节点
        next ← h->Next;    // next 才是真正存储队列第一个元素的节点

        // 悬空一致性检查：确保在读取过程中 Head 没有被其他线程修改
        if (Head ≠ h)
            continue;
    
        // 队列为空检查
        // 如果哨兵节点的下一个是 null，说明队列确实没数据
        if (next == null)
            return EMPTY;

        // 特殊情况处理：队列里只有一个节点，但 Tail 滞后了
        if (h == t) {
            // 尝试帮助入队线程把 Tail 向后推进
            CAS(&Tail, t, next);
            continue; 
        }

        // 准备取出数据
        data ← next->Data;

        // 尝试移动 Head 指针
        if CAS(&Head, h, next)
            break; // 成功出队，跳出循环
    }

    // 返回取出的数据
    return data;
}

加入HP之后的伪代码，改动其实也很简单就是把需要访问的节点用HP保护起来。
但是这里有个坑爹的地方，入队列的时候伪代码里写的是h ← Tail; *hp0 ← h;。这代表着先获尾节点快照，然后加入HP，实际上这是错误的，应该首先加入HP保护，在获取头节点。原因是如果先获取头节点，在头节点被保护之前，就可能失效了。
后续tail->next必定error.

// hp0 和 hp1 是线程私有的风险指针(HP)
Enqueue(data: DataType) {
    node ← NewNode();
    node->Data ← data;
    node->Next ← null;
    while true {
        t ← Tail;
        *hp0 ← t;                     // 标记 t 为正在使用，防止被释放
        if (Tail ≠ t) continue;       // 验证风险指针标记的是否还是最新的 Tail
        next ← t->Next;
        if (Tail ≠ t) continue;
        if (next ≠ null) {            // 发现 Tail 滞后，帮助推进
            CAS(&Tail, t, next);
            continue;
        }
        if CAS(&t->Next, null, node)  // 尝试挂载新节点
            break;
    }
    CAS(&Tail, t, node);              // 尝试将 Tail 移动到新节点
}


Dequeue() : DataType
{
    while (true) {

        // 读取当前 Head（dummy 节点）
        h ← Head;

        // HP0 保护 h
        // 告诉系统：当前线程正在访问 h
        // 在 hp0 清空之前，任何线程都不能释放 h
        *hp0 ← h;

        // 再次确认 Head 没变化
        // 因为在设置 hazard pointer 之前
        // Head 可能已经被别的线程推进
        if (Head ≠ h)
            continue;

        t ← Tail;
        next ← h->Next;
 
        // HP1 保护 next
        // 防止 next 在我们读取 data 前被释放
        *hp1 ← next;

        // 再检查一次 Head 是否变化
        // 如果变化，说明 next 可能已经无效
        if (Head ≠ h)
            continue;
        // 队列为空
        if (next == null)
            return EMPTY;
        // Tail 落后情况（enqueue 未完成）
        // 当前线程帮忙推进Tai
        if (h == t) {
            CAS(&Tail, t, next);
            continue;
        }
        // 读取数据
        // 现在 next 被 hp1 保护，可以安全访问
        data ← next->Data;
        // 尝试推进 Head
        // 谁 CAS 成功谁 dequeue 成功
        if (CAS(&Head, h, next))
            break;
    }
    // 不直接释放内存
    // 因为其他线程可能仍持有 hp0=h
    RetireNode(h);

    return data;
}

代码实现

因为该算法适用于无GC的语言，因此只能选择C/C++/Rust进行实践。原文描述了多种数据结构的HP实现，这里选择最常用的队列完成对论文的复现。 完整代码:pronhub
基于HP的lock-free的队列主要有三种数据结构

  1. 无锁队列–存储业务数据
  2. HP表–访问保护表
  3. 回收队列–延迟回收链表

无锁队列

无锁队列基于 Michael & Scott FIFO 队列，底层是一个带 dummy 节点的单向链表，通过两个原子指针维护：
head：指向 dummy（真实数据在 head->next）
tail：指向队尾附近（允许短暂落后）
节点结构非常简单：

template<typename T>
struct Node {
    std::atomic<Node*> next;
    T data;
};

HP表

HP表是一个全局数组，记录所有线程当前正在访问的节点。

struct HazardPointer {
    std::atomic<std::thread::id> owner;
    std::atomic<void*> ptr;
};

HazardPointer g_hps[MAX_HP];

每个线程会占用少量 slot（通常2个）：

ThreadHP {
    hp0; // protect head/tail
    hp1; // protect next
}

回收队列

普通的队列，可以直接使用STL中的queue。当节点从队列移除后，不能直接 delete，而是进入回收队列。回收队列每个线程私有。达到阈值时会扫描队列中的数据无线程使用的就会被释放。
